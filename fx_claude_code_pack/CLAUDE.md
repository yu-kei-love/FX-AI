# FX AI Project Rules

## 1. プロジェクトの目的
- このプロジェクトは、USD/JPY を対象にした FX AI 自動売買システムを段階的に構築するためのもの。
- 最初の目標は、実弾運用ではなく、データ取得・検証・バックテスト・ペーパートレードまでを安全に完成させること。
- 開発は完全初心者前提で進める。難解な設計より、理解できて動く構成を優先する。

## 2. 最重要方針
- 「動くものを作る」を最優先にする。
- 複雑化しすぎない。
- 現実的でない提案は正直に却下する。
- モデル追加より、検証・執行・コスト・停止条件を優先する。
- 未来リークを絶対に禁止する。
- バックテストの見た目の成績だけで良しとしない。

## 3. 対象範囲
- 初期対象は USD/JPY 1通貨ペアのみ。
- 初期対象は Phase 1〜4 を優先する。
- 実弾自動売買は対象外。最初はバックテストとペーパートレードのみ。

## 4. モデル方針
- Phase 1〜4 の中心は LightGBM + HMM レジーム判定。
- Phase 5 で PPO をポジションサイズ調整用途に限定して導入する。
- TFT / HRL / MARL は最初に実装しない。結果に有意差が出た場合のみ後で検討する。
- ニュース感情分析や過度な外部データ依存は初期では避ける。

## 5. 検証方針
- Triple-Barrier Labeling を採用する。
- Meta-Labeling を採用する。
- Purged K-Fold CV を採用する。
- Walk-Forward 評価を採用する。
- PBO と DSR を実装し、過学習を数値で確認する。
- スプレッド・手数料・スリッページ相当の執行コストを明示的に評価する。

## 6. 執行・運用ルール
- 取引方式は成行のみ。
- スプレッドが閾値を超える場合は見送る。
- 重要経済指標の前後30分は取引停止。
- 取引時間はロンドン時間・NY時間中心。
- 1日最大損失 5% で停止。
- 連続損失 3回で停止・再評価。
- 異常時は自動停止を優先する。

## 7. 実装ルール
- 既存コードがある場合は、その構造を読み取ってから最小変更で適応する。新規作成を前提に壊さない。
- 1タスクごとに、できるだけ小さく安全に変更する。
- 新規コードには必要最小限の日本語コメントを入れる。
- 初心者にも追えるファイル構成を維持する。
- 設定値は可能な限り configs/ に分離する。
- 危険な値をハードコードしない。
- APIキーや秘密情報は .env で扱う。
- Python は 3.11 系を前提にする。
- データ保存の初期手段は CSV または Parquet を優先し、重い DB を最初から導入しない。

## 8. 出力ルール
- 変更したファイルは全文で出力する。
- 変更理由を短く説明する。
- 実行したコマンドを示す。
- テスト結果を示す。
- 残課題があれば明示する。
- エラー時は「原因」「影響」「修正方針」を分けて説明する。

## 9. 停止条件
次の場合は勝手に進めず停止し、ユーザー確認を求める。
- DBスキーマの大きな変更
- 認証・課金・外部送信の追加
- 実弾売買に近づく変更
- 設計書にない大きな方針変更
- 同一タスクで3回以上テスト失敗
- 仕様の曖昧さで実装方針が複数成立する場合
- 必須 API キーや必須環境変数が未設定で、先へ進めない場合

## 10. 開発姿勢
- 「それは後でよい」と判断すべきものは後回しにする。
- 進捗を急ぎすぎない。
- 完璧より、安全で理解できる前進を優先する。
