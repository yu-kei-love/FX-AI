# Acceptance Rules

## 1. 自動で進めてよい変更
- 新規ファイル作成
- 既存仕様内の関数実装
- テスト追加
- 軽微な UI 追加
- 設定ファイル追加
- ログ追加
- README 更新
- requirements / pyproject の軽微な更新
- すでに仕様で明示された API の接続実装

## 2. 必ず停止して確認を求める変更
- DB スキーマの大幅変更
- 認証機構の追加・変更
- 課金関連追加
- 実弾売買につながる変更
- 設計書にない新規外部APIの追加
- 設計書にない大きな仕様変更
- 既存方針を壊すリファクタ
- フォルダ構成の大幅変更

## 3. 失敗時のルール
- 同一タスクで3回連続でテスト失敗したら停止する
- 失敗時は次を報告する
  - 原因
  - 影響範囲
  - 修正案 2〜3 個
- エラーを隠して先へ進まない

## 4. 実装完了時に必ず出すもの
- 変更ファイル一覧
- 変更内容の要約
- 実行したコマンド
- テスト結果
- 次にやるべきタスク
- 残課題

## 5. コード品質ルール
- 初心者でも追いやすい命名を使う
- 不要な抽象化を避ける
- マジックナンバーは config 化する
- API キーは直書きしない
- 未来データ参照を禁止する
- 型ヒントをできる範囲で付ける

## 6. 実装優先順位ルール
- まず Phase 1 を完成させる
- 検証基盤はモデル高度化より先
- 高度モデルは後回し
- 収益化自動投稿機能は初期では実装しない

## 7. 今回の特別ルール
- 現在は Claude Code 向けの初期開発セットアップを最優先する
- まず Task 001 から順番に進める
- 飛ばして高度機能に進まない
- OANDA は Phase 1 の必須 API として扱ってよい
- FRED / Alpha Vantage / NewsAPI は Phase 4 以前には実装を開始しない
