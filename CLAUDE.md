# AI Trading & Prediction Project Rules
# 全プロジェクト統合ルール

## 0. プロジェクト全体の方針

### 最重要原則
- 「動くものを作る」を最優先にする
- 複雑化しすぎない
- 現実的でない提案は正直に却下する
- バックテストの見た目の成績だけで良しとしない
- 未来リークを絶対に禁止する
- 合成データを本番評価に使用することを禁止する
- 「成績が良すぎる」は危険信号として扱う
- モデル追加より、検証・執行・コスト・停止条件を優先する
- 進捗を急ぎすぎない。完璧より、安全で理解できる前進を優先する

### プロジェクト一覧
```
research/
├── fx/          FX自動売買（USD/JPY）
├── boat/        ボートレース予測
├── stock/       株ニュース感情分析
├── crypto/      暗号通貨予測・統計的アービトラージ
└── common/      共通基盤（DB・評価・資金管理）
```

### 共通の停止条件
以下の場合は勝手に進めず停止してユーザー確認を求める：
- DBスキーマの大きな変更
- 認証・課金・外部送信の追加
- 実弾売買に近づく変更
- 設計書にない大きな方針変更
- 同一タスクで3回以上テスト失敗
- 仕様の曖昧さで実装方針が複数成立する場合
- 必須APIキーや必須環境変数が未設定で先へ進めない場合

## 1. FXプロジェクト（research/fx/）

### 目的
- USD/JPYを対象にしたFX AI自動売買システムの構築
- 実弾自動売買は対象外。最初はバックテストとペーパートレードのみ
- 現在フェーズ：ウォークフォワード検証・ペーパートレード

### モデル方針
- 中心：LightGBM + HMMレジーム判定
- Phase5でPPOをポジションサイズ調整用途に限定導入
- TFT・HRL・MARLは有意差が出た場合のみ検討
- ニュース感情分析や過度な外部データ依存は初期では避ける

### 検証方針
- Triple-Barrier Labeling を採用
- Meta-Labeling を採用
- Purged K-Fold CV を採用
- Walk-Forward評価を採用
- PBOとDSRを実装し過学習を数値で確認
- スプレッド・手数料・スリッページを明示的に評価
- Sharpe比を必ず計算すること（目標：≥ 1.0）
- フォールド別詳細（PF・勝率・MDD・Sharpe・総リターン）を必ず保存

### 執行・運用ルール
- 取引方式は成行のみ
- スプレッドが閾値を超える場合は見送り
- 重要経済指標の前後30分は取引停止
- 取引時間はロンドン時間・NY時間中心
- 1日最大損失5%で停止
- 連続損失3回で停止・再評価
- 異常時は自動停止を優先する

### ペーパートレード移行条件（全て満たすこと）
| 指標 | 条件 |
|------|------|
| PF | ≥ 1.3 |
| 最大MDD | ≤ 20% |
| Sharpe比 | ≥ 1.0 |
| トレード数 | ≥ 200 |
| 運用期間 | ≥ 3ヶ月 |

### パラメータ設計の原則
- ハードコードされたパラメータは必ずグリッドサーチで検証する
- 「先に設計した値」と「データで検証した値」を明確に区別する
- 検証していないパラメータは設計書に「未検証」と明記する

### 現在の課題（v3.5）
- DD動的フィルターのパラメータ（×2.0・上限15%）グリッドサーチ実行中
- Fold3（PF=0.83）の原因分析が未完了
- Fold4（16トレード）は統計的に無意味として評価から除外する
- Sharpe比の計算を追加して再実行中

## 2. ボートレースプロジェクト（research/boat/）

### 目的
- ボートレース全場を対象にしたAI予測モデルの構築
- 現在フェーズ：過去データ取得中・特徴量エンジン設計待ち

### 絶対禁止事項
- 合成データの使用（generate_training_dataはNotImplementedErrorにする）
- 実データなしでの評価・成績報告
- 艇番（枠番）を進入コースとして扱うこと

### データ方針
- データソース：BOATRACE.jp のみ
- 保存形式：SQLite（research/boat/data/boatrace.db）
- 目標データ量：2年分・約200,000レース
- スクレイピング間隔：2〜3秒（サーバー負荷配慮）
- 取得済み日付は必ずスキップ（重複防止）

### 最重要特徴量
`course_taken`（実際の進入コース）← 最重要・必須
- 艇番（lane）と混同しないこと
- スタート展示後に確定する
- 15分前の直前取得で取得する

### オッズ取得タイミング（7回・精度最大化）
2時間前・1時間前・30分前・15分前・5分前・1分前・確定
→ 全タイミングをoddsテーブルにtimingカラムで区別して保存

### オッズ急上昇フィルター（必須）
モデルが買いと判断した艇のオッズが15分前→1分前で急上昇している場合：
- 20%以上上昇 → 買いシグナルを完全取り消し
- 10〜20%上昇 → 信頼度を50%カット
- 5〜10%上昇  → 信頼度を20%カット

### モデル方針
- STAGE1：各艇の1着確率（LightGBM + NN + ロジスティック回帰）
- STAGE2：条件付き2着確率
- STAGE3：3連単全120通りの確率計算
- スタッキングで統合

### EV計算（必須）
`EV = 予測確率 × オッズ × 0.75`（控除率25%を必ず考慮）
EV ≥ 1.1 のみ購入候補

### ペーパートレード移行条件
- 実データで3ヶ月以上検証
- ROI > 0%
- 最大MDD < 30%
- カリブレーションのズレ < 10%

### 特徴量設計の原則
- 全特徴量を保存し削除しない（後で重要度スコアで取捨選択）
- カテゴリA〜Jの74特徴量を段階的に追加
- 進入コースベースの交互作用特徴量を使う（艇番ベースは禁止）

### note記録の原則
- 重要なイベントは必ずnote_logger.pyで記録する
- 失敗談・バグ発見・精度変化・設計決定を全て記録
- 毎週月曜日にnote_report_generator.pyを実行してネタを確認

## 3. 株プロジェクト（research/stock/）

### 目的
- ニュース感情分析による株価方向性予測
- 対象：日経225の主要50銘柄（初期）

### 予測の範囲
- 可能：ニュースが出た後の方向性予測
- 不可能：ニュースが出る前の予測
→ この区別を常に意識する

### 感情分析パイプライン
- 第1段階：ルールベース辞書（速度優先・数秒）
- 第2段階：FinBERT（精度優先・数十秒）
- 第3段階：Claude API（文脈理解・数分・複雑なケースのみ）

### 文脈が複雑なケースの処理
以下は第3段階（Claude API）に回す：
- CEO辞任（後任評価が必要）
- 買収・提携（条件次第で逆反応）
- 規制発表（対象範囲が重要）
判断できない場合はスキップする

### 出来高急増フィルター
モデルが買いと判断した銘柄の出来高が直前に急増している場合：
→ 信頼度を下げるまたはスキップ

### 実装の優先順位
- ボートレースが安定してから着手
- FXモデルのペーパートレードが開始してから着手
→ 現時点では設計のみ

## 4. 暗号通貨プロジェクト（research/crypto/）

### 目的
- BTC/USDを中心とした価格方向性予測
- 統計的アービトラージ（中小型アルトコインペア）

### 暗号通貨固有の特徴量
オンチェーンデータ：
- NVT Ratio
- アクティブアドレス数
- 取引所流入/流出量
- クジラの動き

センチメント：
- Fear & Greed Index
- Funding Rate（先物市場の需給）
- Google Trends

### 統計的アービトラージの方針
- 対象：中小型アルトコインペア（時価総額100〜1000億円程度）
- 時間軸：数時間〜1日単位
- 避ける：BTC・ETHの短期（機関投資家の独壇場）
- Zスコア ≥ +2.0 または ≤ -2.0 でシグナル発生
- Zスコア ≥ +4.0 で強制決済（相関が戻らないリスク）

### 実装の優先順位
- ボートレース・FXが安定してから着手
→ 現時点では設計のみ

## 5. 共通基盤（research/common/）

### 共通化する機能
- ニュース収集・感情分析パイプライン
- マクロ経済指標取得（米金利・DXY・VIX等）
- 評価フレームワーク（ROI・MDD・Sharpe比）
- Kelly基準の資金管理
- noteネタ自動記録（note_logger.py）

### 「市場が知っていて自分が知らない情報」への対処
全プロジェクト共通の設計思想：
- ボートレース → オッズ急上昇フィルター
- 株         → 出来高急増フィルター
- FX         → スプレッド急拡大フィルター
- 暗号通貨    → Funding Rate急変フィルター
→ モデルのシグナルより市場の異常な動きを優先する

## 6. 実装ルール（全プロジェクト共通）

- 既存コードがある場合は構造を読み取ってから最小変更で適応する
- 1タスクごとにできるだけ小さく安全に変更する
- 新規コードには必要最小限の日本語コメントを入れる
- 初心者にも追えるファイル構成を維持する
- 設定値はconfigsに分離する
- APIキー・秘密情報は.envで扱う
- Python 3.11系を前提にする
- 変更したファイルは全文で出力する
- エラー時は「原因」「影響」「修正方針」を分けて説明する
- ハードコードしたパラメータは必ず検証する
- 検証していない値はコメントに「未検証」と明記する
- 「それは後でよい」と判断すべきものは後回しにする

---

## 7. note販売ルール

- 全プロジェクトの重要イベントをnote_logger.pyで記録する
- 失敗談・バグ・精度変化・設計決定を全て記録
- 毎週月曜日にnoteネタレポートを確認する
- 「必ず勝てる」等の断言は禁止
- 合成データの結果を実績として掲載しない
- 全記事に免責事項を記載する
